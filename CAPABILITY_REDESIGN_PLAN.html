Context

The user wants to expand ATM's MCP tool surface (informed by homeassistant-ai/ha-mcp) but the existing capability flags panel is already near-unwieldy at 9 flags. Adding ~11 more flags to support new tool categories (discovery, registry CRUD, scenes/helpers, traces, backup, integration management, dashboards, file system, YAML editing) would push the panel past usable.

Two pieces of UX redesign are the right move before expansion:

  1. Personas + per-flag tri-state picker. Replace the wall-of-toggles with role-based presets (Voice Assistant, Automation Builder, etc.) plus a customize view where each capability has three states: Deny / Allow / Confirm.
  2. Generic admin-confirmation gate. When a capability is set to "Confirm," the token's request creates a pending approval record and returns a structured "pending" response. The admin reviews the full diff/args in a new panel page and approves or rejects. Approved actions execute on approval; rejected actions are dropped; unanswered actions expire.

This subsumes the "should we add raw configuration.yaml editing?" question. The confirmation gate makes risky capabilities safer in general — the admin reviews each invocation at the moment it matters, instead of granting blanket permission upfront. (Note on YAML editing specifically: ha-mcp routes through a custom component for technical reasons, not security ones — the HA REST API can't write configuration.yaml. ATM is already a custom integration, so the same _write_utf8_file_atomic path used for automations.yaml would work directly.)

This plan covers only the redesign. Tool expansion is deferred to a follow-up plan that builds on the new capability model.

Goals

  • Replace boolean capability flags with a tri-state model (Deny / Allow / Confirm).
  • Introduce hardcoded personas as presets; allow free-form override.
  • Build a generic pending-approval queue + admin review UI that any "Confirm" capability can route through.
  • Migrate existing storage cleanly. No regressions in existing tools.
  • Keep the model uniform across MCP tools, REST proxy, and admin operations — capability evaluation lives in one place.

Design: Capability Model

Tri-state per capability

Each existing allow_* boolean becomes a cap_* enum with values "deny" | "allow" | "confirm". Capabilities with no realistic confirm semantics (e.g. allow_template_render, allow_log_read) keep deny/allow only — the UI hides the confirm option for them, but storage still uses the enum for uniformity.

Capability tiers (drives UI grouping and which caps offer Confirm):

Two design decisions encoded here, both call-outs in CLAUDE.md after this lands:

  1. deny is overridden by pass-through for non-exempt caps (existing v1 behavior, preserved). Admin who wants to lock down a non-exempt capability for a pass-through token must mark the cap exempt or revoke pass-through.
  2. confirm is honored even for non-exempt caps under pass-through (new). If admin set Confirm, they wanted the gate. Pass-through is a permission-tree convenience, not a license to skip explicit confirmation gates. This means admins can grant pass-through tokens "everything" while still requiring confirm on, e.g., physical_control.

PASS_THROUGH_EXEMPT_FLAGS (in const.py) gets renamed to PASS_THROUGH_EXEMPT_CAPS and updated to use the new cap_* names. Existing exempt set: cap_restart, cap_physical_control, cap_automation_write, cap_script_write, cap_log_read. New caps added later (backup, integration_write, etc.) join the exempt set as appropriate.

Design: Admin-Confirmation Gate

Pending approval lifecycle

LLM calls tool with cap_X = "confirm"
  -> handler creates PendingApproval record
  -> persistent notification fires
  -> response to LLM: { status: "pending_approval", approval_id, expires_at }
  -> LLM moves on or polls

Admin opens /atm panel -> Approvals view
  -> sees pending list with diff/args
  -> approves: original tool runs server-side with saved args
                result stored on approval record
                approval status -> "approved", result available
  -> rejects:  approval status -> "rejected", optional reason
  -> ignores:  TTL expires (default 1 hour), status -> "expired"

LLM polls via new tool get_approval_status(approval_id)
  -> returns current status + result if approved

State machine

                            +---------+
                            | pending |
                            +----+----+
                                 |
        +-------------+----------+-----------+----------------+
        |             |          |           |                |
   admin           admin       TTL        token             admin
   approves        rejects     expires    revoked /         cancels
        |             |          |        kill switch          |
        v             v          v           |                  v
   +--------+    +--------+  +-------+        v             +---------+
   |approved|    |rejected|  |expired|   +---------+        |cancelled|
   +--------+    +--------+  +-------+   |cancelled|        +---------+
                                         +---------+

Terminal states: approved, rejected, expired, cancelled. No further transitions. All transitions out of pending happen under store.async_lock; first writer wins.

Pre-execution re-validation (at admin approve)

Before the approval handler executes the saved action, re-validate all of the following under the lock. Any failure transitions to rejected automatically (not requiring admin reject) with rejected_reason set:

  1. Token still active. If revoked or archived: cancelled with reason token_inactive.
  2. Cap mode still permits execution. If admin changed the cap to deny since the request: rejected with reason capability_denied. If still allow or confirm, proceed.
  3. Entity scope still valid (for tools with entity targets). Re-resolve targets through resolve_intent_entities() and verify each is still WRITE-accessible. If any target moved out of scope: rejected with reason target_out_of_scope. List dropped entities in audit (not in user-facing reason).
  4. Underlying object still exists (for edits and deletes). Confirm the target automation_id, script_id, scene_id, etc. still exists. If deleted by another path: rejected with reason target_missing.
  5. Token rate limit not exhausted. If at limit: rejected with reason rate_limited_at_execution. Better to surface than silently apply rate-limited writes.
  6. Kill switch not engaged. If engaged: cancelled with reason kill_switch.

Only after all checks pass, the saved action runs. The execution audit entry is attributed to the requesting token (so the token's audit log shows what was done in its name) with approved_by_user_id capturing the admin's HA user_id as metadata.

Race conditions and ordering

All status transitions hold store.async_lock. Every concurrency case below resolves cleanly under the lock:

Concurrent operationsResolution
Approve + reject (two admins)First writer wins. Loser receives 409 Conflict with current status.
Approve + TTL expiryLock serializes. If sweeper acquires first, status is expired; admin POST returns 409.
Approve + token revocationRevoke handler scans pending_approvals and sets all token's pendings to cancelled under lock. If admin approve gets lock first, executes; revoke sees no pending. If revoke wins, admin POST returns 409 with status cancelled.
Approve + kill switch engageSame as revocation. Sweeper plus engage path both transition pendings to cancelled.
Approve + cap downgradePre-execute re-validation handles. Admin sees execution result with rejected: capability_denied if applicable.
Two LLM polls during approveRead-only; no contention. Both see the new state once written.
LLM polls + rejectReject completes under lock; subsequent poll returns rejected status.

Idempotency and replay protection

  • POST /api/atm/admin/approvals/{id}/approve and /reject are idempotent on (approval_id, terminal_status). Repeating an approve on an already-approved request returns the cached record with status 200 (no re-execution). Repeating an approve on a rejected request returns 409.
  • Admin endpoints accept an optional If-Match: <ETag> header carrying the current pending record's hash. If the record changed (e.g., transitioned to expired by sweeper) since the admin viewed it, return 412 Precondition Failed. UI auto-refreshes on 412.
  • All admin POST bodies validated against a strict schema; unknown fields rejected.

Tool response when cap mode = confirm

JSON-RPC tool result envelope (matches existing _tool_success shape so clients don't break):

{
  "content": [{
    "type": "text",
    "text": "{\"status\":\"pending_approval\",\"approval_id\":\"appr_abc123\",\"expires_at\":\"2026-05-10T13:30:00Z\",\"review_url\":\"/atm#approvals/appr_abc123\",\"message\":\"This action requires admin approval. The admin has been notified.\"}"
  }],
  "isError": false
}

isError: false because pending is a valid outcome, not a tool error. The LLM should not retry; it should poll get_approval_status later or move on. (REST endpoints that gate on caps return HTTP 202 Accepted with the same JSON body; this is a new outcome alongside the existing 200/4xx codes.)

get_approval_status MCP tool

{
  "name": "get_approval_status",
  "description": "Check the status of a pending approval created by a previous tool call. Returns status (pending/approved/rejected/expired/cancelled), and the result if approved.",
  "inputSchema": {
    "type": "object",
    "properties": {
      "approval_id": {"type": "string"}
    },
    "required": ["approval_id"]
  }
}

Always available — no capability flag required, since the LLM is only ever fetching the status of approvals it created. Tokens can only fetch their own approvals; cross-token reads return 404 (not 403, to avoid an existence oracle). Successful response shape:

{
  "approval_id": "appr_abc123",
  "status": "approved",
  "tool_name": "edit_automation",
  "created_at": "2026-05-10T12:30:00Z",
  "expires_at": "2026-05-10T13:30:00Z",
  "resolved_at": "2026-05-10T12:42:00Z",
  "result": { "automation_id": "atm_xyz", "alias": "Morning routine" },
  "rejected_reason": null
}

Admin endpoints

  • GET /api/atm/admin/approvals?status=pending&token_id=X&limit=50&offset=0 — paginated list. Default returns pending across all tokens.
  • GET /api/atm/admin/approvals/{id} — full record including diff and full args.
  • POST /api/atm/admin/approvals/{id}/approve — body {} or {"note": "..."}; returns the updated record after re-validation and execution.
  • POST /api/atm/admin/approvals/{id}/reject — body {"reason": "..."} (admin-supplied human reason; distinct from auto-rejection reasons); returns updated record.
  • DELETE /api/atm/admin/approvals/{id} — admin cancellation alias for reject with reason: "admin_cancelled".

All require HA admin session (existing require_admin helper). All recorded in audit log with token_id="admin", token_name=f"admin:{user_id}".

Notification dedup and UX

  • Persistent notification ID format: atm_approval_{approval_id}.
  • Notification is dismissed (not just updated) on resolve — admin shouldn't see stale entries.
  • Multiple pending approvals from the same token: each gets its own notification.
  • HA event atm_approval_requested fires on creation; atm_approval_resolved fires on terminal status with the resolution kind. This lets admins build their own automations (e.g., push to mobile, Slack notify).

Per-tool diff builders

Every approval record carries a diff field with a uniform envelope. The kind discriminates how the frontend renders it. Each Confirm-eligible tool implements a _build_diff_<tool>(args, token, hass) -> dict function returning this envelope at request time:

{
  "kind": "yaml_diff" | "config_diff" | "service_preview" | "system_action" | "file_write",
  "summary": "Edit automation 'morning_routine'",
  "target": { "type": "automation", "id": "atm_xyz", "label": "Morning routine" },
  "before": "...",
  "after": "...",
  "preview": { /* kind-specific structured detail */ }
}

Diff size policy

  • Inline before/after capped at 100KB combined per record. Records exceeding the cap store a content hash and the first/last 50 lines; full content is fetchable via GET /api/atm/admin/approvals/{id}/diff/full (admin-only).
  • service_preview.resolved_entity_ids capped at 500 entries inline; over that, paginated.
  • All caps live in const.py (MAX_DIFF_INLINE_BYTES, MAX_PREVIEW_ENTITY_IDS) so they can be tuned without code changes.

Storage location

Add pending_approvals: [] array to .storage/atm.json. Cap to MAX_PENDING_APPROVALS = 100 per token (reject new requests with 429 if at cap). Cleanup task runs every 5 minutes (reuse the existing async_track_time_interval cadence) to expire stale entries.

Generic capability check helper

Replace scattered if not token.allow_X: return 403 checks with a single helper in helpers.py:

async def evaluate_capability(
    cap_name: str,
    token: TokenRecord,
    hass: HomeAssistant,
    *,
    diff_builder: Callable[[], dict] | None = None,
    args: dict,
    tool_name: str,
    request_id: str,
) -> CapabilityResult:
    """Returns Allow / Deny / PendingApproval(approval_id)."""

Each tool handler calls this once at the top:

result = await evaluate_capability("cap_yaml_edit", token, hass, ...)
if result.is_deny:
    return _tool_error("Forbidden")
if result.is_pending:
    return _tool_pending(result.approval_id, result.expires_at)
# proceed with action

Approval execution path

When the admin approves, the backend re-runs the tool's server-side action using the saved args. To make this safe and reusable, factor each tool handler into:

  1. _tool_<name>(args, token, hass) — entry point: capability check + dispatch.
  2. _execute_<name>(args, token, hass) — pure side-effect path that assumes capability is satisfied.

_execute_<name> is what the approval handler calls. The original _tool_<name> calls it after a successful Allow check. This is the only refactor required for existing tools that gain a Confirm option.

Notifications and events

  • Persistent notification fired on creation: "ATM: Token '{name}' requested {tool_name}. Review at /atm#approvals."
  • HA event atm_approval_requested fires for automation hooks (e.g. push notification to admin's phone).
  • HA event atm_approval_resolved fires on approve/reject/expire with status.

Why pull model (LLM polls) over hold-the-connection

Holding the SSE/HTTP connection until admin approves means a tool call can hang for an hour. Most clients time out. Pull model: handler returns immediately with pending_approval; LLM either tells the user "this needs admin approval, I'll check back" or polls via get_approval_status. Cleaner for everyone.

Audit Log Additions

The existing audit.py circular buffer entry shape (per CLAUDE.md): request_id, timestamp, token_id, token_name, method, resource, outcome, client_ip.

New fields

  • cap_mode_at_check: "deny" | "allow" | "confirm" | null — what the cap was set to when the check ran. null for entries where no cap was checked (e.g., entity-tree-only checks).
  • approval_id: string | null — set when the entry is the original request that created a pending approval, OR the entry that records execution after admin approval.
  • executed_via_approval: bool — true on the audit entry that records the action's actual execution after admin approval; false on the original gate entry and on entries that didn't go through the gate.
  • pass_through_at_check: bool — was the token in pass_through mode when checked.
  • approved_by_user_id: string | null — set on the execution entry to the admin's HA user_id.

New outcome values

  • pending_approval — original request created a pending approval; nothing executed yet.
  • approval_executed — a previously-pending approval was approved and the action ran.
  • approval_rejected — admin rejected (or auto-reject from re-validation failure).
  • approval_expired — TTL elapsed.
  • approval_cancelled — token revoked, kill switch engaged, or admin cancelled.

Lifecycle example

A single edit_automation that goes through the gate produces these audit entries:

Steptoken_idoutcomeapproval_idexecuted_via_approvalapproved_by_user_id
1. Original tool callmorning_assistantpending_approvalappr_xyzfalsenull
2. Admin approve actionadminallowedappr_xyzfalse(the admin's id)
3. Action executionmorning_assistantapproval_executedappr_xyztrue(the admin's id)

Step 3 is attributed to the requesting token so the token's audit log shows what was done in its name. Admin auth is preserved as metadata via approved_by_user_id. Admin actions on the gate (step 2) live in the admin audit stream.

If the admin rejects instead, only steps 1 and 2 occur, with step 2's outcome denied and a separate cross-reference entry on the requesting token's audit stream with outcome approval_rejected.

mcp_view.py token-self-summary endpoints

The endpoints that return the token's effective capabilities for LLM context (lines 2136, 2191, 2263) need updated formulas. New helper effective_cap(token, cap_name) -> "deny" | "allow" | "confirm":

def effective_cap(token, cap_name) -> str:
    raw = getattr(token, cap_name)  # "deny" | "allow" | "confirm"
    if cap_name in PASS_THROUGH_EXEMPT_CAPS:
        return raw  # exempt: pass-through doesn't affect
    if token.pass_through:
        if raw == "confirm":
            return "confirm"  # design decision: confirm honored under pass-through
        return "allow"  # deny and allow both -> allow under pass-through
    return raw

This formula applies anywhere the current code computes flag or token.pass_through. Use it consistently across REST and MCP serialization paths.

Storage Migration

STORAGE_VERSION bumps from 1 to 2. Implement async_migrate_func on the Store:

async def _migrate_storage(old_version: int, old_data: dict) -> dict:
    if old_version == 1:
        for token in old_data.get("tokens", []) + old_data.get("archived_tokens", []):
            for old_key in [
                "allow_automation_write", "allow_script_write", "allow_config_read",
                "allow_template_render", "allow_restart", "allow_physical_control",
                "allow_service_response", "allow_broadcast", "allow_log_read",
            ]:
                new_key = old_key.replace("allow_", "cap_")
                token[new_key] = "allow" if token.pop(old_key, False) else "deny"
            token.setdefault("persona", "custom")
        old_data.setdefault("pending_approvals", [])
    return old_data

Out of Scope (deferred to follow-up plan)

This section captures the full critical analysis of ha-mcp's tool surface so the follow-up plan can pull from a concrete idea bank. Tools listed below are NOT being implemented in this plan; they wait for the capability redesign to land first.

Tier A — ha-mcp parity (recommended subset)

Tools worth importing roughly as-is once the new capability model is in place:

  • Discovery / search: search_entities, deep_search, get_overview. Highest LLM-UX value. Behind cap_search.
  • Registry reads: list_areas, list_floors, list_zones, list_devices, get_device. Permission-scoped read-only views of HA's registries. Behind cap_registry_read.
  • Scene CRUD: create_scene, edit_scene, delete_scene, list_scenes. Mirrors existing automation/script CRUD pattern. Behind cap_scene_write.
  • Helper CRUD: create_helper, edit_helper, delete_helper, list_helpers. Wraps input_*, counter, timer. Behind cap_helper_write. Open question on which helper types support service-based create vs. WebSocket-only.
  • Traces: get_automation_traces. Behind cap_traces.
  • Diagnostics: get_system_health, check_config. Behind cap_diagnostics.
  • Risky-tier system tools: create_backup, restore_backup (cap_backup); set_integration_enabled, HACS install/search (cap_integration_write); dashboard CRUD (cap_lovelace_write); file system access scoped to www//themes//custom_templates/ (cap_filesystem); raw configuration.yaml edits (cap_yaml_edit). Each of these benefits enormously from the Confirm gate — they are exactly the cases the gate was designed for.

Tier A.1 — ha-mcp tools to skip (questionable value)

Reasoning recorded so the follow-up plan doesn't relitigate:

  • ha_install_mcp_tools, ha_manage_custom_tool — meta-tools for installing more MCP tools. Pointless in ATM since ATM is itself the MCP server.
  • ha_report_issue — bug-report submission. No reason an LLM operating someone's home should file bugs upstream.
  • ha_get_blueprint, ha_import_blueprint — blueprints are templates. LLMs are more likely to write automation YAML directly than browse a catalog. Marginal utility for the surface cost.
  • ha_manage_addon — add-on install/remove is admin-console work. Wrong abstraction for token-scoped LLM access; risky and rarely needed.
  • ha_config_set_dashboard_resource, ha_config_list_dashboard_resources — dashboard internals at extreme granularity. LLMs almost never edit these. Skip even if cap_lovelace_write lands.
  • ha_eval_template overlapping ha_get_state — states('light.x') and get_state('light.x') return the same primitive. ATM keeps both render_template and get_state, but the redundancy is acceptable since each has its own use case (template eval vs. structured state). Just don't add a third path.
  • ha_get_camera_image — useful in principle, but binary over MCP/SSE means base64-encoded chunks; large payload risk and many clients can't render images inline. If added later, do so carefully with a size cap and an opt-in flag.

Tier B — improvements over ha-mcp (worth doing rather than copying)

Each is a deliberate departure from ha-mcp's design, motivated by a specific shortcoming:

  • Dry-run mode for service calls + standalone dry_run_service — call_service (and ha-mcp's bulk_control) execute immediately. Adding an optional dry_run: true argument that returns resolved targets and predicted state changes (without executing) gives LLMs a way to reason about effects before committing. Pairs naturally with the admin-confirmation gate: a Confirm-gated tool can produce a dry-run-style preview for the admin to review. A standalone dry_run_service(domain, service, args) tool is a useful explicit form too.
  • whatif(entity_id, hypothetical_state) — walks loaded automations to show which would fire if X became Y. Predictive analysis without invoking. Much more powerful than just reading automation YAML.
  • get_history returning transitions — current implementation (and ha-mcp's) returns raw state-per-second dumps that bloat context. Adding a mode that returns only state transitions (or auto-resamples to a granularity hint) is far more useful for LLM reasoning. Make it the default; preserve the raw mode as opt-in.
  • get_automation_traces summarized output — the raw HA trace structure is dense. Adding a summary: true mode that highlights the failing condition or service call (with the surrounding context only) yields much better "why did this fail?" answers. Keep raw available for debug.
  • search_entities with state-aware filters — beyond fuzzy name match, support filters like device_class, state value, "stale > N hours," "unavailable," and area scoping. Discovery should be state-aware, not just text-aware.
  • deep_search with cross-references — instead of plain text matches, return structured relationships: "this automation references entity X," "this entity is referenced by automations Y, Z." Becomes useful for impact analysis, not just keyword search.
  • get_state with related context — optional flag to include "same device," "same area," and "referenced by" data. Reduces follow-up calls from typical LLM workflows.
  • config_set_automation with referenced-entity validation — schema-valid is not the same as runtime-valid. Validate that referenced entities exist AND are accessible to the calling token before saving. Prevents broken automations from being created with placeholder entities.
  • watch_entity(entity_id, timeout) and subscribe_event(event_type, timeout) — bounded subscriptions over the existing SSE channel. Lets LLMs verify the effect of an action they just took without polling, and lets them reactively wait for events without persisting an automation. ATM's SSE infrastructure already exists (heartbeat, connection registry, revocation termination) so the wiring is mostly free. Time-boxed (e.g. max 60 seconds, configurable cap) to avoid hanging connections.

Tier C — ATM-only opportunities (impossible in ha-mcp's model)

These are tools that only make sense in a system with scoped tokens, capability flags, and per-token audit history:

  • get_capability_summary() — the token introspects: "here is what you can read, here is what you can control, here is what is currently gated by Confirm, here are your active personas, here are your rate limits." Lets the LLM orient itself at session start without trial-and-error tool calls. Particularly valuable in the new tri-state cap model since the LLM can plan around Confirm-gated capabilities rather than discovering them mid-flow. No equivalent exists in ha-mcp because ha-mcp has no scoped-token concept.
  • get_audit_summary(window?) — per-token recent activity. "What have I done in the last hour?" Returns the calling token's own audit entries (request_id, tool, outcome, resource), filtered to that token only — no leakage across tokens. Useful for LLM self-correction ("did my last service call succeed?") and for end-user transparency. Ties directly to the existing audit.py circular buffer.

Tier C.1 — additional missing tools worth considering

Logged here for completeness; less central than the four ranked items above but recurring useful primitives:

  • describe_entity(entity_id) — single call returns: what it is, what services act on it, recent state summary, which automations/scripts/scenes reference it. Comprehension primitive.
  • get_relationships(object_id) — generalised reverse-dependency lookup across entities, automations, scripts, scenes, helpers, dashboards.
  • recent_activity(window) — structured summary of "what changed in the last N minutes" at the home level (not the token level — that's get_audit_summary). Fired automations, called services, state changes grouped by area. The "catch me up" primitive.
  • schedule_action(when, action) — first-class one-off scheduled action. Currently you have to create-then-delete an automation; awkward.
  • set_temporary_state(entity, state, duration) — assert a state for N minutes then revert. The "turn this on for 10 minutes" primitive.
  • compare_state(t1, t2, entity_ids) — diff two snapshots. "What changed while I was away?"
  • describe_area(area_id) — area summary: entities by domain, aggregate states, devices, occupancy. Area-level context primitive.
  • validate_config(yaml_blob, type) — validate automation/script/scene YAML structurally without saving. Decouples schema check from commit; useful as a precursor when admin-confirmation gating is enabled.
  • find_available_actions(entity_id | area_id) — given a target, what services could be invoked? Useful for empty-state UX and exploration.

Sequencing for the follow-up plan

The three tiers compose naturally:

  1. Tier A delivers familiar coverage and validates the new capability model under realistic load.
  2. Tier B improves the LLM-reasoning surface where ATM can do better than ha-mcp, leveraging the now-stable foundation.
  3. Tier C exploits ATM's structural advantage with introspection tools that no other HA MCP server can offer.

All of the above benefit from this redesign and become much easier to add cleanly afterward.
